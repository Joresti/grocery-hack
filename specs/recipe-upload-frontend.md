Feature: Recipe Upload — Frontend Specification

This spec defines the frontend UX for user-uploaded recipes. It covers where
recipes appear on the landing page, the RecipeFormModal (create/edit), URL
import, ingredient entry, the "My Recipes" section, and the recipe detail
modal with deal-enriched pricing.

Relies on: specs/recipe-upload.md (behavioral), api-contract.yaml (endpoints),
docs/design/style-guide.md (tokens), docs/design/component-tree.md (tree).

Key layout decision: Community recipes (other users' public recipes) are mixed
into the swipe deck in Dream Meal Matching. The current user's own recipes
live in a dedicated "My Recipes" section directly below Dream Meal Matching,
with an inline "Add a Recipe" CTA.


# Landing Page Section Order (updated)

  Header
  Primary Actions (Optimize, Feeling Lucky, View Shopping List, My Staples)
  Absurd Deal Alert (conditional)
  Your Recipes on Sale (conditional — user recipes matching deals)
  My Liked Meals (conditional)
  Dream Meal Matching (swipe cards — includes community recipes in deck)
  **My Recipes + Add a Recipe (NEW — always visible for auth'd users)**
  Store Meal Deal List + Shopping List
  Notable Deals


Feature: Community Recipes in Swipe Deck

Scenario: Public user recipes appear in the swipe deck
Given another user has published a recipe (is_public: true)
And the planner has matched it against current deals
Then it should appear in the swipeableMeals array from GET /landing
And it should render as a normal meal card in Dream Meal Matching

Scenario: Attribution on community recipe card
Given a meal card in the swipe deck originated from a user recipe
Then a small attribution line should appear below the price badge:
  | element             | style                                  |
  | initials circle     | 24px, random pastel bg, white text, Sora 600 0.7rem |
  | "Shared by {name}"  | Inter 400, 0.8rem, textMuted           |

Scenario: System-generated meal has no attribution
Given a meal card was generated by the system (not a user recipe)
Then no attribution line should appear


Feature: My Recipes Section — Landing Page

Scenario: Section placement
Given the user is authenticated
Then a "My Recipes" section should appear directly below Dream Meal Matching
And directly above the Store Meal Deal List

Scenario: Section not shown for anonymous users
Given the user is not authenticated
Then the "My Recipes" section should not appear

Scenario: Section header
Given the "My Recipes" section is visible
Then the header should contain:
  | element           | style                                  |
  | section title     | "My Recipes" Sora 700, 1.15rem         |
  | recipe count      | "(X)" next to title, textMuted, Sora 400 0.9rem |
  | "+ Add" button    | right-aligned, primary color, Inter 500 0.85rem, no bg |

Scenario: User has recipes — card display
Given the user has saved one or more recipes
Then recipes should appear as compact horizontal-scroll cards (like Liked Meals preview)
And each card should display:
  | element                      | style                              |
  | recipe name                  | Inter 600, 0.9rem, text color, 2 lines max, ellipsis |
  | ingredient count             | "X ingredients" Inter 400, 0.8rem, textMuted |
  | deal match badge (if matched)| pill, greenBadgeBg, greenBadgeText, "X on sale" Sora 600 0.75rem |
And cards should have: white bg, 16px radius, card shadow, 16px padding, 140px min-width
And horizontal gap between cards should be 12px
And the scroll container should hide the scrollbar but allow horizontal swipe/drag

Scenario: User taps a recipe card
Given the user taps one of their recipe cards
Then the RecipeDetailModal should open for that recipe (deal-enriched if matches exist)

Scenario: User has no recipes — empty state
Given the user has no saved recipes
Then instead of cards, the section should show a single CTA card:
  | element           | content                                |
  | icon              | line-art recipe/book icon, textMuted, 32px |
  | message           | "Save your favorites" Inter 500, 0.9rem |
  | sub-message       | "We'll watch for deals on your ingredients" Inter 400, 0.8rem, textMuted |
  | button            | "Add a Recipe" primary button, pill, full card width |
And the CTA card should have the same card styling (white bg, 16px radius, shadow)
And it should be centered, max-width 280px

Scenario: Add a Recipe button behavior
Given the user taps "+ Add" in the section header or the empty state CTA
Then the RecipeFormModal should open in create mode


Feature: RecipeFormModal — Layout and Structure

Scenario: Modal shell
Given the RecipeFormModal opens
Then it should use the standard ModalOverlay pattern:
  | property         | mobile                    | desktop              |
  | position         | slides up from bottom     | center-aligned       |
  | max height       | 90vh                      | 85vh                 |
  | max width        | 100%                      | 520px                |
  | border radius    | 20px top corners          | 16px all corners     |
  | backdrop         | rgba(0,0,0,0.5)          | rgba(0,0,0,0.5)     |
  | animation        | spring (damping 25, stiffness 300) | fade + scale |
  | close            | X button top-right, Escape key, backdrop tap |
And the body should scroll vertically with 24px padding
And document body overflow should be hidden while modal is open

Scenario: Modal header
Given the RecipeFormModal is open in create mode
Then the header should display "New Recipe" in Sora 700, 1.25rem
And a close (X) icon button in the top-right corner (44x44px touch target)

Given the RecipeFormModal is open in edit mode
Then the header should display "Edit Recipe" instead

Scenario: Modal content sections (top to bottom)
Given the RecipeFormModal is open
Then the form should contain these sections in order:
  | section              | required |
  | URL Import           | no       |
  | Recipe Name          | yes      |
  | Ingredients          | yes      |
  | Steps                | no       |
  | Details (grid)       | no       |
  | Dietary Tags         | no       |
  | Notes                | no       |
  | Share toggle         | no       |
  | Action buttons       | —        |
And sections should have 24px vertical spacing between them


Feature: RecipeFormModal — URL Import Section

Scenario: Import section layout
Given the RecipeFormModal is open in create mode
Then the first section should be a URL import bar
And it should contain a single-line text input and an "Import" button side by side
And the input placeholder should be "Paste a recipe URL..."
And the Import button should be a small primary button (teal bg, white text, pill shape)
And the Import button should be disabled when the input is empty

Scenario: Import section not shown in edit mode
Given the RecipeFormModal is open in edit mode
Then the URL import section should not be visible

Scenario: User triggers import
Given the user has pasted a URL into the import field
When the user taps "Import" or presses Enter
Then the Import button should show a spinner (16px, white, rotating)
And the input should become read-only during the request
And the rest of the form should remain visible below but dimmed (opacity 0.5)

Scenario: Import succeeds
Given the import request returns successfully
Then the form fields should auto-populate with extracted data:
  | field       | maps to                    |
  | recipe name | name input                 |
  | ingredients | ingredient rows            |
  | steps       | step rows                  |
And a success toast should appear: "Recipe imported — review and save"
And the toast should use the standard pattern (fade in, Y translation, 1.5s)
And the URL import bar should collapse to show the source URL as a small link
And all auto-populated fields should be editable

Scenario: Import fails
Given the import request fails or returns no data
Then an inline error message should appear below the import bar
And the message should read: "We couldn't read that recipe — try adding it manually"
And the message should be in Inter 400, 0.85rem, danger color (#DC2626)
And the error should dismiss after 5s or when the user starts typing in any field
And the manual form below should remain fully interactive (opacity restored to 1)

Scenario: Import partial data
Given the import extracts some but not all fields (e.g., name and ingredients but no steps)
Then only the extracted fields should auto-populate
And empty fields should remain blank for manual entry
And the success toast should still appear


Feature: RecipeFormModal — Recipe Name Input

Scenario: Name field layout
Given the RecipeFormModal is open
Then a "Recipe Name" label should appear (Inter 500, 0.85rem, textMuted)
And below it a single-line text input:
  | property       | value                              |
  | placeholder    | "e.g., Grandma's Beef Stew"        |
  | background     | white                              |
  | border         | 1.5px solid border color           |
  | border-radius  | 12px                               |
  | padding        | 12px horizontal, 12px vertical     |
  | font           | Inter 400, 0.9rem                  |
  | min-height     | 44px                               |
  | focus state    | border transitions to primary      |
  | max length     | 100 characters                     |


Feature: RecipeFormModal — Ingredient Entry

Scenario: Ingredient section header
Given the RecipeFormModal is open
Then an "Ingredients" section header should appear
And it should contain the label "Ingredients" (Inter 600, 0.95rem) on the left
And an "+ Add" text button on the right (primary color, Inter 500, 0.85rem)

Scenario: Ingredient row layout
Given the ingredient section is visible
Then each ingredient row should contain three inputs in a horizontal row:
  | input     | flex   | placeholder     |
  | name      | 1      | "Ingredient"    |
  | quantity  | 60px   | "Qty"           |
  | unit      | 70px   | "Unit"          |
And a remove button (X icon, 20px, textMuted, 44x44px touch target) on the right
And rows should have 12px vertical spacing between them
And all inputs should use the standard input style (white bg, 1.5px border, 10px radius)

Scenario: Initial state
Given the RecipeFormModal opens in create mode
Then one empty ingredient row should be pre-populated
And the remove button should be hidden when only one row exists

Scenario: Adding ingredients
When the user taps "+ Add"
Then a new empty ingredient row should appear at the bottom with a subtle fade-in
And the new row's name input should auto-focus

Scenario: Removing ingredients
Given there are two or more ingredient rows
When the user taps the remove button on a row
Then that row should be removed with a 150ms fade-out
And if only one row remains, its remove button should hide

Scenario: Keyboard flow
Given the user is typing in an ingredient name field
When the user presses Tab
Then focus should move to the quantity field of the same row
When the user presses Tab again
Then focus should move to the unit field of the same row
When the user presses Tab from the last row's unit field
Then a new ingredient row should be added and its name field should auto-focus

Scenario: Mobile ingredient layout (< 480px)
Given the viewport width is less than 480px
Then ingredient rows should stack: name input on first line (full width)
And qty + unit inputs side by side on second line with remove button


Feature: RecipeFormModal — Steps Entry

Scenario: Steps section header
Given the RecipeFormModal is open
Then a "Steps" section header should appear
And it should contain the label "Steps" (Inter 600, 0.95rem) on the left
And an "+ Add Step" text button on the right (primary color, Inter 500, 0.85rem)

Scenario: Step row layout
Given the steps section has rows
Then each step row should contain:
  | element             | style                                    |
  | number badge        | 24px circle, primary bg, white text, Sora 600 0.75rem |
  | text input          | flex 1, standard input style, placeholder "Describe this step..." |
  | remove button       | X icon, 20px, textMuted, 44x44px target  |
And rows should have 12px vertical spacing
And step numbers should auto-update when rows are removed

Scenario: Initial state
Given the RecipeFormModal opens in create mode
Then the steps section should start empty (no rows)
And a subtle hint should appear: "Optional — add cooking instructions" (Inter 400, 0.8rem, textMuted)

Scenario: Adding and removing steps
When the user taps "+ Add Step"
Then step row 1 should appear with auto-focus on the text input
And the hint text should disappear
When the user removes all steps
Then the hint text should reappear


Feature: RecipeFormModal — Details Grid

Scenario: Details grid layout
Given the RecipeFormModal is open

And it should contain a 3-column grid (2-column on mobile < 480px):
  | field        | input type | placeholder | unit label |

  | Cook Time    | number     | "0"         | "min"      |
  | Servings     | number     | "4"         | —          |
And each field should have a label above (Inter 500, 0.85rem, textMuted)

And inputs should be 60px wide with the unit label inline to the right

Feature: RecipeFormModal — Action Buttons

Scenario: Create mode buttons
Given the RecipeFormModal is in create mode
Then the bottom should have a sticky action bar (white bg, top border, 16px padding)
And it should contain a single full-width "Save Recipe" button
And the button should be primary style (teal bg, white text, pill shape, Inter 600)
And the button should be disabled (opacity 0.5) until name is non-empty AND at least one ingredient has a name

Scenario: Edit mode buttons
Given the RecipeFormModal is in edit mode
Then the action bar should contain two buttons side by side:
  | button          | style                               | position |
  | Delete Recipe   | danger outline (white bg, danger border, danger text) | left |
  | Save Changes    | primary (teal bg, white text)       | right    |

Scenario: Save in progress
Given the user taps "Save Recipe" or "Save Changes"
Then the button text should change to "Saving..." with a 16px spinner
And the button should be disabled
And all form fields should become read-only

Scenario: Save succeeds
Given the save request completes successfully
Then a success toast should appear: "Recipe saved!"
And the modal should close after a 300ms delay
And the landing page data should refetch (invalidate landing query)
And the new recipe should appear in the "My Recipes" section

Scenario: Save fails
Given the save request fails
Then an error message should appear above the action bar
And the message should read: "Something went wrong — please try again"
And the message should be in danger color
And the button should re-enable

Scenario: Delete confirmation
Given the user taps "Delete Recipe" in edit mode
Then a confirmation dialog should appear (centered, 280px wide, 16px radius, white bg):
  | element           | content                               |
  | title             | "Delete Recipe?" Sora 600 1.1rem      |
  | body              | "This can't be undone." Inter 400 0.9rem textMuted |
  | cancel button     | "Cancel" secondary style, left        |
  | confirm button    | "Delete" danger style (danger bg, white text), right |
When the user taps "Delete"
Then the recipe should be deleted, modal should close, toast: "Recipe deleted"


Feature: RecipeFormModal — Validation

Scenario: Required field validation
Given the user taps "Save Recipe" and the recipe name is empty
Then the name input border should turn danger color
And "Recipe name is required" should appear below the input (danger, 0.8rem)

Given the user taps "Save Recipe" and no ingredients have names
Then the ingredient section header should turn danger color
And "Add at least one ingredient" should appear below the section header

Scenario: Inline validation timing
Then validation errors should only appear after the first save attempt
And once shown, errors should clear as soon as the field becomes valid


Feature: Your Recipes on Sale — Deal Cards

Scenario: Section placement (unchanged from existing spec)
Given the landing response includes recipe_alerts with matches
Then the "Your Recipes on Sale" section appears between Absurd Deal Alert and My Liked Meals
And the section header: "Your Recipes on Sale" Sora 700, 1.15rem

Scenario: Recipe deal card layout
Given a recipe alert card is displayed
Then the card should have: white bg, 16px radius, 20px padding, card shadow
And the card content:
  | element                       | style                                   |
  | recipe name                   | Sora 600, 1rem, text color              |
  | "X of Y ingredients on sale"  | Inter 400, 0.85rem, textMuted           |
  | this week price               | Sora 700, 1.1rem, primary color         |
  | regular price                 | Inter 400, 0.85rem, textMuted, strikethrough |
  | savings badge                 | pill, greenBadgeBg, greenBadgeText, "Save $X.XX" Sora 600 0.8rem |

Scenario: Card interaction
When the user taps a recipe deal card
Then the RecipeDetailModal should open for that recipe with deal-enriched pricing
And on hover, the card shadow should intensify (0.3s ease)


Feature: Recipe Detail Modal — Deal-Enriched View

Scenario: Modal opens from recipe deal card or My Recipes card
Given the user taps a recipe card (from deal alerts or My Recipes section)
Then a RecipeDetailModal should open using the standard ModalOverlay pattern

Scenario: Modal header
Given the RecipeDetailModal is open
Then the header should show:
  | element           | style                                  |
  | recipe name       | Sora 700, 1.25rem                      |
  | source badge      | "Your Recipe" pill, primaryLight bg, primary text, 0.75rem |
  | close button      | X icon, top-right, 44x44px             |

Scenario: Ingredient list with deal enrichment
Given the recipe has ingredients
Then each ingredient should display as a row:
  | element           | style                                  |
  | ingredient name   | Inter 500, 0.9rem                      |
  | quantity + unit   | Inter 400, 0.85rem, textMuted, right side |

Scenario: Ingredient on sale
Given an ingredient matches a current deal
Then the row should have a primaryLight background (full-width)
And a deal price: Sora 600, 0.85rem, primary color
And regular price next to it: strikethrough, textMuted, 0.8rem
And small store name: Inter 400, 0.75rem, textMuted

Scenario: Ingredient not on sale
Given an ingredient does not match any deal
Then the row should have no background highlight
And a small "pantry" label in textMuted, italic, 0.75rem

Scenario: Savings summary
Given the ingredient list is complete
Then a summary row should appear below with a top border:
  | element                | style                              |
  | "Est. total this week" | Inter 500, 0.9rem                  |
  | deal total price       | Sora 700, 1.1rem, primary          |
  | "Regular ~$XX.XX"      | Inter 400, 0.85rem, textMuted, strikethrough |
  | savings amount         | greenBadgeText, Sora 600           |

Scenario: Steps section
Given the recipe has steps
Then a "Steps" section should appear below ingredients
And each step: number badge (24px circle, primary bg, white text) + text (Inter 400, 0.9rem)
And 16px vertical spacing between steps

Scenario: Details and notes
Given the recipe has optional details or notes
Then details should appear as horizontal pills:
  | example          | style                                  |
  | "15 min prep"    | primaryLight bg, primary text, 0.8rem  |
  | "30 min cook"    | same                                   |
  | "Serves 4"       | same                                   |
And notes should appear in a styled block (bg color background, 12px radius, 16px padding, Inter 400 0.85rem textMuted)

Scenario: Edit button
Given the modal shows the current user's own recipe
Then a secondary "Edit Recipe" button should appear at the bottom
When tapped, RecipeFormModal opens in edit mode and RecipeDetailModal closes


Feature: Responsive Behavior

Scenario: Mobile (< 480px)
Given the viewport is less than 480px wide
Then RecipeFormModal should be full-width, slide up from bottom
And ingredient rows should stack: name on top, qty + unit on second line
And the details grid should be 2 columns
And action buttons should be full-width stacked (delete above save in edit mode)
And My Recipes cards should be horizontally scrollable, single row

Scenario: Tablet (480px - 768px)
Given the viewport is 480px-768px wide
Then ingredient rows should remain horizontal
And the details grid should be 3 columns

Scenario: Desktop (> 768px)
Given the viewport is greater than 768px
Then the modal should center at 520px max-width
And all layouts remain horizontal


Feature: Loading and Empty States

Scenario: Form loading (edit mode)
Given the RecipeFormModal opens in edit mode and recipe data is being fetched
Then a skeleton loader should match the form layout
And use the standard shimmer animation (primaryLight to white, 1.5s)

Scenario: Import loading
Given the user triggered a URL import
Then only the import section shows a spinner
And the form below dims but remains visible

Scenario: My Recipes section loading
Given the landing data is loading
Then the My Recipes section should show 3 skeleton cards (shimmer, 140px wide each)


Feature: Accessibility

Scenario: Keyboard navigation
Given the RecipeFormModal is open
Then all interactive elements should be reachable via Tab
And focus should be trapped within the modal while open
And Escape should close the modal
And the first input (URL import or recipe name) should auto-focus on open

Scenario: Screen reader support
Given the RecipeFormModal is open
Then the modal should have role="dialog" and aria-modal="true"
And the title should be referenced by aria-labelledby
And remove buttons should have aria-label="Remove ingredient" or "Remove step"
And the share toggle should have aria-label="Share recipe with community"
And error messages should use aria-describedby on their associated inputs
